## Server Scripts

Files in this folder are operational or diagnostic utilities.

They are **not** part of the request-serving runtime path.

### Current intent

- `replay-pipeline-suite.ts`
- `replay-pipeline.ts`
- `replay-strategic-context-suite.ts`
- `validate-production-readiness-real.ts`

### Data-driven pipeline improvement (Step 0–1)

- `npm run data-driven:coverage` — JSON report of recommendation snapshot coverage vs settled-replay export filters (`--lookback-days`, `--out-json`). The `currentRuntime` block separates official `v10-hybrid-legacy-g` export-eligible rows with non-empty `decision_context` from legacy prompt versions, empty prompt versions, and official-prompt rows missing replay context.
- `npm run data-driven:prompt-adoption -- --lookback-days 14 [--max-recent-rows 50] [--out-json <file>] [--out-md <file>]` — recent recommendation canary for prompt-version adoption before rows settle. Use it to verify whether saved rows are being stamped with official `v10-hybrid-legacy-g` and non-empty `decision_context`; this complements, but does not replace, settled replay coverage.
- `npm run data-driven:pipeline-liveness -- --lookback-hours 336 [--job-name check-live-trigger] [--max-recent-rows 25] [--out-json <file>] [--out-md <file>]` — DB-only liveness audit for the live recommendation writer. It combines `job_run_history`, `audit_logs` pipeline events, and `recommendations` recency to distinguish prompt adoption gaps from scheduler/job inactivity.
- `npm run data-driven:current-runtime-no-save -- --lookback-hours 336 [--max-samples 50] [--out-json <file>] [--out-md <file>]` — official-prompt no-save diagnostics from `audit_logs`. Use it when `pipeline-liveness` proves `v10-hybrid-legacy-g` is active but no recommendations are saved; it breaks down model no-bets, market resolution, policy warnings, evidence mode, save integrity, and recent samples.
- `npm run data-driven:current-runtime-blocked-selection -- --lookback-hours 336 [--max-rows 1000] [--stake-percent 1] [--out-json <file>] [--out-md <file>]` — reviews official-prompt `PIPELINE_MATCH_ANALYZED` rows where the model had a non-empty selection but runtime policy/save gates produced no push/save. It joins `matches_history` and uses deterministic settlement rules to estimate counterfactual P/L for supported markets; this is audit evidence only.
- `npm run data-driven:check-current-runtime-blocked-selection-gates -- --config current-runtime-blocked-selection-gates.json` — checks a saved blocked-selection review for minimum settled coverage and optional required canonical-market thresholds. Copy from `current-runtime-blocked-selection-gates.example.json`; a passing result means the pocket is worth shadow observation, not production promotion.
- `npm run data-driven:check-prompt-adoption-gates -- --config recommendation-prompt-adoption-gates.json` — checks a saved prompt-adoption report for minimum recent rows, latest-row recency, and official prompt adoption. Copy from `recommendation-prompt-adoption-gates.example.json`; use this after deploy before treating production rows as evidence for `v10-hybrid-legacy-g`.
- `npm run data-driven:check-coverage-gates` — `--config recommendation-snapshot-coverage-gates.json` checks a saved coverage report for minimum export-eligible rows, current-runtime-ready rows, and maximum legacy/empty-context rates. Use this after `data-driven:coverage` when you need to prove whether settled production rows exist for the official prompt.
- `npm run data-driven:replay-batch` — writes `replay-work/data-driven-runs/<runId>/`: `coverage.json`, `run-spec.json`, scenarios, eval reports, and (unless `--no-post-summarize`) `replay-vs-original.json` + `cases-flat.csv` (variant 0). Unless `--no-post-segment-hotspots`, also writes `segment-hotspots.json` (variant 0: minute band × market family rollups) and `segment-action-plan.json` (quality blockers plus reviewable block/cap/review candidates). Default `--llm mock`; real: `--llm real --allow-real-llm`. `--apply-replay-policy` for production parity after parse. If `LIVE_ANALYSIS_*_PROMPT_VERSION` env vars are unset, eval falls back to code default `LIVE_ANALYSIS_PROMPT_VERSION` (`v10-hybrid-legacy-g`).
- `npm run data-driven:improvement-run` — preset batch for analysis/improvement loops: `--lookback-days 14`, `--limit 50`, `--max-scenarios 30`, mock LLM, `odds recorded`, `--apply-replay-policy`. Same outputs as `replay-batch`; read `replay-vs-original.json`, `segment-hotspots.json`, `segment-action-plan.json`, `eval-summary.md`, and `cases-flat.csv` first to prioritize prompt or segment policy changes.
- For non-overlapping replay audit chunks, pass `--offset N` together with `--limit`; for example `--limit 40 --offset 40`.
- `npm run data-driven:improvement-run-real` — same shape as above but **`--llm real`** (needs `GEMINI_API_KEY`, smaller cap: 15 scenarios, 1.2s delay). Writes **`llm-cache/*.json`** per case (`aiText`, `prompt`, `selection`) even when `should_push` is false — use these to audit refusals (e.g. “no usable odds”) vs snapshot shape. Increase `--max-scenarios` only deliberately (cost/latency).
- `npm run data-driven:fast-hotspot-real` — quick hotspot gate (real LLM): 10-day window, 80 scenarios, faster delay. Best for short inner-loop checks right after policy/prompt edits.
- `npm run data-driven:fast-smoke-real` — wider smoke gate (real LLM): 10-day window, 150 scenarios, medium delay. Use after hotspot pass to catch regressions before full 400.
- `npm run data-driven:fast-gate-real` — sequential `fast-hotspot-real` + `fast-smoke-real` as a single command.
- `npm run data-driven:summarize-vs-original` — `--cases-json <eval-cases.json> [--out-json] [--out-csv] [--csv-variant-index N]` for Step 2 delta vs production outcomes on an existing run.
- `npm run data-driven:aggregate-cases -- --cases-json <run-a/eval-cases.json> --cases-json <run-b/eval-cases.json> [--out-json <file>]` aggregates multiple real/mock replay chunks into one deduped `eval-cases.json`, preserving source order and reporting duplicate scenario names.
- `npm run data-driven:normalize-diagnostics -- --cases-json <eval-cases.json> [--out-json <file>]` rewrites existing eval cases with current replay diagnostic labels, for example splitting empty-selection no-bets from true selected-market resolution failures without re-calling the LLM.
- `npm run data-driven:check-gates` — Step 3: `--config data-driven-replay-gates.json` (copy from `data-driven-replay-gates.example.json`, set `deltaPath` + `promptVersion` + thresholds); exit 1 if replay-vs-original metrics regress.
- `npm run data-driven:segment-hotspots` — `--cases-json <eval-cases.json> [--variant-index 0] [--out-json ...] [--min-settled N] [--min-staked N]` for segment rollups on an existing eval run (same logic as batch `segment-hotspots.json`).
- `npm run data-driven:check-segment-gates` — Step 4b: `--config data-driven-segment-gates.json` (copy from `data-driven-segment-gates.example.json`, set `hotspotPath` + optional `promptVersion` + per-segment `rules`); exit 1 if hotspot metrics breach thresholds.
- `npm run data-driven:check-quality-gates` — Step 4c: `--config data-driven-quality-gates.json` (or CI config). Reads `segment-action-plan.json` and fails when replay has too much provider coverage mismatch or replay context gap before prompt/policy tuning.
- `npm run data-driven:verify-gates-ci` — same checks as CI: `ci-baselines/data-driven-gates/gates-delta.ci.json` + `gates-segment.ci.json` (static JSON, UTF-8). Repo root `npm run verify:ci` runs server+client typecheck/tests plus this.
- **GitHub Actions (on demand):** `.github/workflows/data-driven-db-coverage.yml` — Postgres service, `npm run migrate`, then `data-driven:coverage` writing `replay-work/ci-artifacts/snapshot-coverage.json` and uploading artifact `data-driven-snapshot-coverage` (empty DB is valid; counts are zero).
- **Runtime segment blocklist** (optional): set `SEGMENT_POLICY_BLOCKLIST_PATH` to a JSON file like `segment-policy-blocklist.example.json` (`minuteBand::marketFamily` keys). Live pipeline policy will block matching recommendations.
- **Runtime segment stake cap** (optional): `SEGMENT_POLICY_STAKE_CAP_PATH` → `segment-policy-stake-cap.example.json` (`caps` map). Matching segments get `min(modelStake, cap)` with warning `POLICY_WARN_SEGMENT_STAKE_CAP` (blocklist still wins if both apply).
- **Line Ladder Patience** (default on): `LINE_PATIENCE_ENABLED` (`false` to disable). Optional `LINE_PATIENCE_CONFIG_PATH` → `line-patience-policy.example.json`. Runs after AI parse, before `applyRecommendationPolicy` — see [line-ladder-patience-spec.md](../../../docs/line-ladder-patience-spec.md).
- **Thesis watch** (default on, requires LLP): `THESIS_WATCH_ENABLED`, `THESIS_WATCH_TTL_MINUTES`. Persists LLP deferrals and promotes on the next cycle without a second LLM call — see spec Phase 2.
- `npm run data-driven:suggest-segment-blocklist` — `--hotspots-json <segment-hotspots.json>` prints JSON `{ segmentKeys }` (default: union top 8 worst-accuracy + top 8 worst-ROI rows). Tighten with `--max-accuracy`, `--max-roi`, `--worst-accuracy-top 0` to disable a source, `--out-json <file>` to write.
- `npm run data-driven:action-plan -- --hotspots-json <segment-hotspots.json> [--eval-cases-json <eval-cases.json>] [--out-json <file>]` — builds a reviewable quality action plan from segment hotspots. With `--eval-cases-json`, it also writes `qualityBlockers` (decision diagnostics, market-resolution status, top warnings, and representative unresolved/policy-blocked cases), which is important when replay has zero actionable pushes. Severe underperformers become blocklist candidates, moderate weak segments become stake-cap candidates, and production-loss segments that replay still wants are flagged for prompt/provider review. It does not modify runtime policy files; copy accepted candidates into `SEGMENT_POLICY_BLOCKLIST_PATH` / `SEGMENT_POLICY_STAKE_CAP_PATH` deliberately.
- `npm run data-driven:policy-experiment -- --cases-json <eval-cases.json> [--out-json <file>] [--stake-cap experiment_id=N]` runs replay-only counterfactuals for narrow policy-blocked model selections. It estimates stake-capped PnL only when replay and original markets match; it does not modify runtime policy.
- `npm run data-driven:check-policy-experiment-gates -- --config replay-policy-experiment-gates.json` checks a replay-only policy experiment report against cohort size, candidate count, selected pockets, ROI/PnL, and loss-reintroduction thresholds. Required experiment rules can also enforce pocket-level `minWinCount`, `minOriginalWinsRescued`, `minTotalPnlPercent`, `minRoiOnStaked`, `maxLossCount`, and `maxOriginalLossesReintroduced`. Copy from `replay-policy-experiment-gates.example.json`; this is for shadow/replay validation only, not runtime persistence.
- `npm run data-driven:policy-shadow-suite -- --lookback-days 14 --settlement-lookback-days 30 --max-rows 1000 --stake-percent 1 --out-dir replay-work/runtime-policy-shadow-suite/<runId>` writes the runtime shadow reports (matched, skipped-neighbor, matched settlement, skipped settlement), Phase 3 readiness gates, and `manifest.json` in one folder. This is the easiest periodic operator command after shadow telemetry has been live for a while.
- Runtime shadow operator runbook: [docs/runtime-shadow-operator-runbook.md](../../../../docs/runtime-shadow-operator-runbook.md) documents when to run the liveness/no-save/blocked-selection/shadow-suite sequence, which files to read first, what pass/fail means, and when promotion is forbidden.
- `npm run data-driven:policy-shadow-report -- --lookback-days 14 --max-rows 1000 [--out-json <file>] [--out-md <file>]` reads `PIPELINE_POLICY_SHADOW_CANDIDATE` audit events and aggregates runtime shadow candidates by pocket, canonical market, minute band, score state, confidence band, market-resolution status, and market availability bucket. This is telemetry only; use it after shadow pockets have run live for a while before considering any production promotion.
- `npm run data-driven:policy-shadow-skipped-report -- --lookback-days 14 --max-rows 1000 [--out-json <file>] [--out-md <file>]` reads `PIPELINE_POLICY_SHADOW_SKIPPED` audit events for policy-blocked model selections that did not match a configured shadow pocket. It includes confidence and market-resolution breakdowns; use this to review nearby risk areas such as low-price BTTS, late Over 1.5 after minute 75, or late Under 2.5 level-score candidates before adding new pockets.
- `npm run data-driven:policy-shadow-skipped-settlement -- --lookback-days 30 --max-rows 1000 [--stake-percent 1] [--out-json <file>] [--out-md <file>]` joins skipped-neighbor shadow events to `matches_history`, settles supported markets, and reports counterfactual P/L by canonical market, skipped reason, league segment, and team segment. Use it to prove whether nearby skipped candidates were mostly policy-saved losses or missed winners before adding new shadow pockets.
- `npm run data-driven:check-policy-shadow-skipped-settlement-gates -- --config runtime-policy-shadow-skipped-settlement-gates.json` checks a saved skipped-neighbor settlement report against sample size, settled-rate, loss, P/L, ROI, and optional required market/skipped-reason thresholds. Copy from `runtime-policy-shadow-skipped-settlement-gates.example.json`; this is evidence for whether to design a new shadow pocket, not an automatic runtime change.
- `npm run data-driven:policy-shadow-settlement -- --lookback-days 30 --max-rows 1000 [--out-json <file>] [--out-md <file>]` joins runtime shadow audit events to `matches_history`, settles supported markets with deterministic rules only, and reports shadow P/L by pocket, canonical market, league segment, and team segment. Rows without match history or rule coverage remain unresolved; this command does not call Gemini and does not persist anything.
- `npm run data-driven:check-policy-shadow-settlement-gates -- --config runtime-policy-shadow-settlement-gates.json` checks a saved runtime shadow settlement report against sample size, settled-rate, loss, P/L, ROI, and required-pocket thresholds. Copy from `runtime-policy-shadow-settlement-gates.example.json` and run it only after `data-driven:policy-shadow-settlement`; a passing gate is promotion evidence, not an automatic production policy change.
- `npm run data-driven:check-policy-shadow-readiness-gates -- --config runtime-policy-shadow-readiness-gates.json` checks combined matched/skipped telemetry and settlement artifacts for Phase 3 hard no-promote rules: sample, settled coverage, loss/P&L/ROI, match/league/team/market concentration, market-resolution gaps, and evidence-mode contamination. Copy from `runtime-policy-shadow-readiness-gates.example.json`; pass means human review only, not Phase 4 rollout.
- Phase 4 controlled promotion is configured only through `RUNTIME_POLICY_PROMOTION_*` env vars. Readiness gates do not enable it automatically; `RUNTIME_POLICY_PROMOTION_EVIDENCE_ACK=ready_for_human_review`, pocket allowlist, rollout percentage, owner, and kill switch posture must be set deliberately.
- Replay quality attribution fields in `eval-cases.json` / `cases-flat.csv`: `provider_coverage` means the recorded provider snapshot did not have the requested historical line; `replay_context_gap` means replay lacked context such as performance memory; `pre_llm_blocked` means the auto-LLM eligibility firewall skipped the call before any model response; `model_policy_mismatch` means the model proposed a bet in a hard-policy zone already exposed by runtime preflight prompt context; `hard_policy_gate` means odds resolved but a remaining post-parse policy gate blocked. Treat provider/context rows as diagnostics before changing prompt or segment policy.
- `replay-vs-original.json` now includes `opportunityTradeoff` (original directional winner recall/miss rate and original directional loser avoidance/replay rate). `segment-action-plan.json` includes `qualityBlockers.opportunityRecall` and `qualityBlockers.modelSelectedPolicyBlocked`; use these before loosening prompt or policy so recall gains are weighed against avoided historical losses.
- Runtime save integrity: recommendation saves now record `saveIntegrityStatus`, `saveProviderCoverageStatus`, `saveMarketResolutionStatus`, `saveMappedOdd`, `savedSelection`, and `savedBetMarket` in `decision_context`. If the save candidate cannot be proven against canonical provider odds, the pipeline audits `RECOMMENDATION_SAVE_BLOCKED_PROVIDER_COVERAGE` and does not create a recommendation row.
- `npm run data-driven:memory-rebuild` rebuilds `recommendation_performance_memory` from trusted settled recommendations. Run before replay analysis when `segment-action-plan.json` shows `memory_no_history`; it prints source rows, aggregate groups, total samples, and reliable groups.
- `npm run provider:coverage-audit --prefix packages/server -- --lookback-days 180 --limit 500` replays stored `provider_odds_samples.normalized_payload` through the current coverage classifier and canonical odds builder. Use `--fail-on-mismatch` for a hard gate and `--out-json <path>` to persist the report. This proves whether stored provider coverage flags are consistent with canonical markets without spending real provider quota. The report separates raw provider availability (`raw_has_*`) from canonical tradable availability (`canonical_has_*`).
- `npm run provider:real-sample --prefix packages/server -- --max-fixtures 3 --max-api-calls 12 --out-json replay-work/audit/<runId>/provider-samples/real-provider-samples.json` spends real API-Football quota to sample live, near-kickoff, and finished fixtures for one date. Defaults to today's date in `TIMEZONE`, summary payloads only, and all calls routed through `football-api.ts`. Use `--date YYYY-MM-DD`, `--no-live`, `--no-near`, `--no-finished`, `--no-stats`, `--no-events`, `--no-live-odds`, `--no-prematch-odds`, or `--full-payloads` for targeted audits. Use `--iterations N --interval-ms MS` for bounded repeated live-day sampling; `--max-api-calls` applies across all iterations.
- `npm run provider:coverage-backfill --prefix packages/server -- --lookback-days 180 --limit 1000` dry-runs a recomputation of `provider_odds_samples.coverage_flags` from stored normalized payloads. Add `--apply` only when you intentionally want to correct historical sample flags used by Ops Monitoring. Backfilled flags keep legacy `has_*` raw semantics and add explicit `raw_has_*` / `canonical_has_*` fields.

Use these only for:

- replay
- validation
- benchmarking
- diagnostics

Do not treat them as required production workflow steps.
If the application only works after manually running one of these scripts, that is a product/runtime bug and should be fixed in the server jobs or routes instead.

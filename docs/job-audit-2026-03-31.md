# Job Audit - 2026-03-31

## Scope

This audit covers all background jobs and job-like maintenance flows currently present in `packages/server/src/jobs` plus the shared scheduler, progress, and housekeeping paths they depend on.

Reviewed runtime paths:

- `scheduler.ts`
- `job-progress.ts`
- `fetch-matches.job.ts`
- `refresh-live-matches.job.ts`
- `sync-reference-data.job.ts`
- `enrich-watchlist.job.ts`
- `update-predictions.job.ts`
- `check-live-trigger.job.ts`
- `refresh-provider-insights.job.ts`
- `auto-settle.job.ts`
- `re-evaluate.job.ts`
- `expire-watchlist.job.ts`
- `purge-audit.job.ts`
- `integration-health.job.ts`
- `health-watchdog.job.ts`

## Executive Summary

The job system is functionally rich, but it is still not fully consistent at the orchestration layer.

The most important problem is not a single broken job. It is that the scheduler semantics, job result semantics, and housekeeping semantics are not aligned with the operational promises the system is trying to make:

- live jobs are configured at `5s`, but the scheduler does not guarantee a `5s` effective cadence
- several jobs report success metrics that can be technically true at the code level but operationally misleading
- housekeeping exists, but retention policy and storage control are still uneven across tables and job outputs
- some jobs are carrying too many responsibilities, so failures and retries have wider side effects than necessary

If the goal is predictable live freshness, quota control, and operational trust, the next phase should start with scheduler hardening and result/metric normalization before more job-specific refactors.

## Priority Findings

### P1 - Live cadence is not actually guaranteed by the scheduler

Evidence:

- `scheduler.ts` uses `setInterval(() => runJob(job), job.intervalMs)`.
- Single-concurrency jobs immediately return when `job.running` is already `true`.
- `refresh-live-matches` and `check-live-trigger` are both configured at `5s` in `config.ts`.

What this means:

- The effective cadence for these jobs is not `intervalMs`.
- The effective cadence is `max(intervalMs, actual runtime)` plus timer jitter.
- If a live run takes `6-8s`, one or more intervals are silently dropped.
- For a live product where `10s` lag is already too much, this is a real user-facing risk.

Affected jobs:

- `refresh-live-matches`
- `check-live-trigger`
- indirectly `health-watchdog`, because its health model assumes interval-driven regularity

Impact:

- stale live scores or delayed pipeline reactions during provider slowness
- false confidence from the UI because the configured interval looks stricter than the real runtime behavior
- difficult root-cause analysis because skipped intervals are not reported as lag

Recommended fix:

- replace interval-driven execution with self-rescheduling execution per job
- record `scheduled_at`, `started_at`, `completed_at`, `duration_ms`, and `lag_ms`
- for freshness-critical jobs, keep at most one pending rerun instead of silently dropping intervals
- expose lag metrics in the jobs UI and watchdog logic

### P1 - Redis outage currently means job outage for almost every scheduled job

Evidence:

- `scheduler.ts` `acquireLock()` returns `false` on Redis errors
- every registered scheduled job currently uses `concurrency = 1`
- single-concurrency jobs do not run without the Redis lock

What this means:

- Redis is not only an optimization or coordination layer
- Redis is currently a hard availability dependency for nearly the entire scheduler

Impact:

- if Redis is unavailable, match ingest, live refresh, settlement, housekeeping, and monitoring jobs all stop running
- the system can appear partially alive while the background engine is effectively down

Recommended fix:

- define per-job lock policy explicitly:
- `strict_distributed_lock_required` for jobs where duplicate execution is unacceptable
- `degraded_single_instance_allowed` for jobs that can safely run on the current instance if Redis is down
- at minimum, alert immediately when Redis blocks a critical job run instead of only logging it

### P1 - Scheduler state can become stale and mislead monitoring

Evidence:

- `persistState()` writes `running` to Redis with no TTL
- `restoreState()` does not restore or clear `running`
- `getJobsStatus()` trusts `state.running === '1'` from Redis over local state
- `health-watchdog` uses `getJobsStatus()` for overdue and stuck detection

What this means:

- a crashed process can leave behind a permanent `running=1` flag
- the jobs UI and watchdog can treat a dead job as "running"
- this causes false stuck alerts or masks a real overdue condition

Impact:

- low trust in the jobs page
- noisy or misleading watchdog alerts
- harder incident triage because operator state is contaminated by stale metadata

Recommended fix:

- move from a static `running` flag to heartbeat-based run state with TTL
- clear stale state on startup if the owning instance heartbeat is gone
- separate `last_started_at` and `last_completed_at`

### P1 - Monitoring coverage does not match the real critical path

Evidence:

- `health-watchdog.job.ts` critical set includes:
- `fetch-matches`
- `check-live-trigger`
- `auto-settle`
- `expire-watchlist`
- `update-predictions`
- `enrich-watchlist`
- it does not include `refresh-live-matches`

What this means:

- the live score refresh path can stall while the watchdog remains green
- this is now inconsistent with the system's own live-freshness policy

Impact:

- user-facing live freshness can degrade with no direct alert

Recommended fix:

- add `refresh-live-matches` to the critical job set
- consider classifying jobs by criticality and freshness budget instead of a hardcoded list

### P1 - Job result payloads are too large and create storage and observability noise

Evidence:

- `scheduler.ts` writes the full job `result` object into audit metadata
- `job-progress.ts` stores `JSON.stringify(result)` in Redis
- `check-live-trigger` returns `pipelineResults`
- `re-evaluate` returns potentially large `discrepancies`

What this means:

- some jobs are emitting operational summaries mixed with large per-item payloads
- Redis progress keys and `audit_logs` can grow much faster than necessary
- this undermines housekeeping by generating avoidable churn

Impact:

- audit log bloat
- larger Redis payloads for the jobs page
- harder operator debugging because high-signal summary metrics are buried in large blobs

Recommended fix:

- standardize job result contracts to summary-only payloads
- move large per-item details to explicit drill-down tables or ad hoc reports
- cap serialized progress result size

## Important Findings

### P2 - `fetch-matches` is carrying too many responsibilities

Current responsibilities in one job:

- fixture ingest
- live stat enrichment
- finished-match archive
- full `matches` table refresh
- operational watchlist backfill
- watchlist kickoff sync
- top-league auto-add
- favorite-team auto-add
- adaptive next-run control

Why this matters:

- one late-stage failure marks the whole job failed even if the core ingest already committed
- retries repeat side effects that are not core to match ingestion
- performance tuning becomes coarse because all work shares one schedule and one status

Impact:

- harder incident isolation
- partial success patterns are hidden behind a single job status
- favorite-team auto-add can become a nonlinear cost center because it performs per-user existence checks inside nested loops

Recommended fix:

- split into:
- `ingest-fixtures`
- `archive-finished-matches`
- `sync-watchlist-metadata`
- `auto-add-top-league-watchlist`
- `auto-add-favorite-team-watchlist`
- keep adaptive polling attached only to the ingest step

### P2 - `refresh-provider-insights` metrics are inconsistent with runtime behavior

Evidence:

- `fixtureCached` starts as `fixtures.length` and never increments meaningfully
- `oddsRefreshed` is always returned as `0`
- the scheduler description says "live and followed games"
- the code explicitly excludes live matches and only prewarms non-live candidates

Impact:

- operators can draw the wrong conclusion about quota use and cache effectiveness
- job output cannot be trusted for tuning or capacity planning

Recommended fix:

- return separate counters for:
- `candidateCount`
- `liveSkipped`
- `fixtureCacheHits`
- `fixtureRefreshes`
- `statisticsRefreshes`
- `lineupsRefreshes`
- `predictionsRefreshes`
- `standingsRefreshes`
- `oddsRefreshes`
- make the scheduler description match the actual behavior

### P2 - `sync-reference-data` can overstate refresh success

Evidence:

- `sync-reference-data.job.ts` counts every fulfilled `refreshLeagueTeamsDirectoryNow()` call as refreshed
- `league-team-directory.service.ts` can fall back to stale local rows on provider failure and still fulfill

Impact:

- the job can report successful team-directory refreshes even when the provider refresh failed
- reference-data staleness is harder to detect

Recommended fix:

- change the service return type to include `source: refreshed | stale_fallback | cache`
- count stale fallback separately from actual provider refresh
- consider using stale-aware refresh rules instead of unconditional force refresh for every active league

### P2 - `enrich-watchlist` can starve the long tail under backlog

Evidence:

- sequential processing
- `2s` delay between items
- heavy AI and web grounding work per item
- `25m` soft deadline
- `1h` scheduler interval

Impact:

- some eligible matches can repeatedly miss enrichment before kickoff
- quality becomes dependent on queue shape rather than business priority

Recommended fix:

- replace in-memory force flag with persistent queue control
- store `next_enrich_at` and `attempt_count` in durable state
- prioritize by kickoff proximity, top-league status, and previous failures
- keep a strict per-run work budget and continue fairly on the next run

### P2 - `update-predictions` has the same durability issue as `enrich-watchlist`

Evidence:

- `forceNext` is process-local
- predictions are processed sequentially with `200ms` delay

Impact:

- manual force refresh is not reliable in multi-instance deployment
- large watchlists can drift behind the intended update window

Recommended fix:

- move force triggers to Redis or DB
- add a durable `prediction_refresh_requested_at`
- batch by kickoff and freshness instead of a plain sequential sweep

### P2 - Settlement logic is centralized only at the rule engine layer, not at the orchestration layer

Evidence:

- `auto-settle` and `re-evaluate` both:
- fetch missing fixtures
- archive finished matches
- fetch match statistics
- derive settlement score
- call `settleMatch()`
- persist recommendation and AI performance updates

Impact:

- duplicated orchestration logic makes settlement behavior drift more likely
- fixes must be repeated in two places
- provider and archive access patterns are harder to reason about

Recommended fix:

- create a `settlement-orchestrator` with shared stages:
- `load-history`
- `hydrate-missing-results`
- `hydrate-settlement-stats`
- `build-settlement-context`
- `apply-outcomes`
- make `re-evaluate` consume the same orchestration with a different selection strategy

### P2 - Housekeeping is useful, but still too monolithic

Evidence:

- `purge-audit.job.ts` runs every purge in a single `Promise.all`
- a single failure fails the whole job
- `VACUUM ANALYZE` only covers a subset of the tables being purged

Impact:

- large daily delete bursts can increase lock contention or I/O spikes
- partial success is not reported cleanly
- some high-churn tables may still accumulate bloat despite being purged

Recommended fix:

- split housekeeping into phases:
- `log-retention`
- `provider-samples-retention`
- `analytics-retention`
- `recommendation-slimming`
- `maintenance-vacuum`
- use `Promise.allSettled` plus per-phase reporting
- trigger vacuum/analyze based on per-table deletion thresholds, not one global total

### P2 - Retention policy for `user_recommendation_deliveries` is unresolved

Evidence:

- `purge-audit.job.ts` explicitly does not purge `user_recommendation_deliveries`
- `recommendation-deliveries.repo.ts` already exposes `purgeOldDeliveries()`

Impact:

- the codebase contains two competing signals:
- "never globally purge"
- "generic age-based purge exists"
- this is a policy inconsistency and a future maintenance trap

Recommended fix:

- define a formal retention policy for delivery history
- if retention must be user-lifecycle driven, remove or isolate generic age-based purge helpers
- if age-based retention is acceptable for some environments, wire it explicitly with configuration and documentation

## Lower-Severity Findings

### P3 - `expire-watchlist` under-reports what it changed

Evidence:

- the repo returns `Math.max(expiredMatchIds.length, monitoredResult.rowCount ?? 0)`

Impact:

- the job output does not tell operators how many subscriptions were removed versus how many monitored rows were removed

Recommended fix:

- return structured counts for:
- `expiredSubscriptions`
- `refreshedSubscriberCounts`
- `deletedMonitoredMatches`

### P3 - Scheduler registration is not idempotent

Evidence:

- `jobs` is a module-global array
- `startScheduler()` appends registrations every time it is called

Impact:

- low in the current boot path because the app appears to call it once
- still risky for tests, manual boots, or future lifecycle hooks

Recommended fix:

- prevent double registration or clear the registry before registering

### P3 - Test coverage is weaker at the orchestration layer than at the individual job layer

Evidence:

- many job tests exist
- there is no dedicated scheduler test suite covering:
- Redis outage behavior
- stale `running` state
- interval drift
- duplicate `startScheduler()` calls
- during this audit, the targeted job suite also exposed one stale unit test:
- `src/__tests__/auto-settle.test.ts` fails because its mock for `../lib/football-api.js` no longer matches the current import graph

Impact:

- refactors can keep job internals green while breaking the orchestration contract

Recommended fix:

- add scheduler-focused tests first
- repair `auto-settle.test.ts` so unit coverage tracks the current runtime path

## Recommended Remediation Phases

### Phase 1 - Scheduler and observability hardening

- redesign scheduler cadence for freshness-critical jobs
- add lag and duration telemetry
- move to heartbeat-based run state
- add Redis outage alerting policy
- include `refresh-live-matches` in watchdog coverage
- shrink job result payloads to summary-only

Expected impact:

- improves live freshness predictability
- improves trust in monitoring
- reduces audit and Redis noise

### Phase 2 - Split `fetch-matches` into smaller operational boundaries

- separate ingest from watchlist side effects
- isolate favorite-team auto-add into its own job or post-ingest stage
- make non-core stages soft-fail without marking ingest as failed

Expected impact:

- easier incident isolation
- more stable retries
- lower chance that non-core automation blocks core ingest confidence

### Phase 3 - Build a real settlement orchestration layer

- share hydrate/archive/stats/settlement stages between `auto-settle` and `re-evaluate`
- record when AI fallback is used and why
- monitor unresolved settlement backlog explicitly

Expected impact:

- less logic drift
- lower provider duplication
- clearer settlement SLA

### Phase 4 - Durable backlog control for prematch jobs

- replace process-local force flags
- use durable next-at scheduling
- prioritize by kickoff and business value

Expected impact:

- fewer silent misses before kickoff
- more predictable prematch quality under load

### Phase 5 - Policy-complete housekeeping

- turn housekeeping into phased retention
- define delivery-history retention formally
- review per-table vacuum/analyze thresholds

Expected impact:

- better storage control
- lower maintenance risk
- less surprise from hidden retention drift
